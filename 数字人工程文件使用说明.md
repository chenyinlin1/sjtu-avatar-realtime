## 数字人工程文件使用说明

### 一、启动相关

#### 1. 启动前需要修改的配置

如果是把整个工程复制到新的 AutoDL 实例上，启动前先不要急着运行程序。

可以先把它理解成“搬家”：代码搬过去了，但门牌号、门、钥匙这些东西要先确认。

最少需要确认下面几个配置。

##### 1.1 新实例公网地址

这个地址是别人从浏览器里访问数字人的地址，例如：

```text
https://新的实例地址:8443
```

它最重要，因为**音色克隆**会用这个地址告诉百炼：

> 你去这个网址下载刚刚录好的声音。

如果这里还写着旧实例地址，音色克隆就会跑去旧机器下载录音，容易失败。

需要修改的文件：

```bash
/root/autodl-tmp/chatrobot_rebuild_workspace/OpenAvatarChat/start_realtime_human.sh
```

找到类似下面的位置：

```bash
case "$PORT" in
  6006)
    DEFAULT_PUBLIC_URL="https://u848390-a11e-6f8e472d.cqa1.seetacloud.com:8443"
    ;;
```

把里面的旧地址改成新实例的公网地址：

```bash
DEFAULT_PUBLIC_URL="https://u848390-ab99-ddff46b2.cqa1.seetacloud.com:8443"
```



##### 1.2 程序内部端口

现在默认内部端口是 `6006`。

还是在这个文件里：

```bash
start_realtime_human.sh
```

找到：

```bash
PORT="${PORT:-6006}"
```

如果新实例还是用 `6006`，这里不用改。

如果要换成别的端口，例如 `6008`，这里也要一起改。

##### 1.3 YAML 配置文件里的端口

需要确认这个文件：

```bash
config/chat_with_openai_compatible_bailian_cosyvoice_flashhead_6006.yaml
```

找到：

```yaml
service:
  host: "0.0.0.0"
  port: 6006
```

这里的 `port` 必须和 `start_realtime_human.sh` 里的 `PORT` 一样。

简单建议：如果没有特殊需要，新实例也继续用 `6006`，这样最省事。

##### 1.4 stop 脚本里的端口和配置文件

停止程序的脚本也要保持一致。

需要确认这个文件：

```bash
stop_realtime_human.sh
```

找到：

```bash
CONFIG_FILE="${CONFIG_FILE:-config/chat_with_openai_compatible_bailian_cosyvoice_flashhead_6006.yaml}"
PORT="${PORT:-6006}"
```

这里的 `CONFIG_FILE` 和 `PORT` 要和启动脚本一致。

否则启动和停止可能找的不是同一个程序。

##### 1.5 AutoDL 端口映射或 nginx 转发

这个不是代码里的配置，而是服务器外面的“入口”配置。

要确认：

```text
外面访问的公网端口  ->  服务器里面程序实际监听的端口
```

例如：

```text
https://新的实例地址:8443  ->  127.0.0.1:6006
```

如果外面访问 `8443`，但里面程序跑在 `6006`，就必须保证 AutoDL 端口映射或 nginx 把 `8443` 转到 `6006`。

否则用户能看到公网地址，但页面打不开。

##### 1.6 TURN 配置

视频通话有时候需要 TURN 中转服务器。

需要确认这个文件：

```bash
config/chat_with_openai_compatible_bailian_cosyvoice_flashhead_6006.yaml
```

找到：

```yaml
turn_config:
  turn_provider: turn_server
  urls:
    - "stun:stun.l.google.com:19302"
    - "turn:115.159.81.115:3478?transport=udp"
    - "turn:115.159.81.115:3478?transport=tcp"
  username: "turnuser"
  credential: "Abc_2026_Turn_x9Kp7"
```

如果新实例继续使用这台 TURN 服务器，就不用改。

如果换成自己的 TURN，就要把 `115.159.81.115`、`username`、`credential` 一起换掉。

##### 1.7 API Key

目前启动脚本里可能会写默认的第三方服务 Key，例如：

```bash
DASHSCOPE_API_KEY
BOCHA_API_KEY
```

如果不准备改成环境变量形式，也至少要确认：

- 新实例是否继续使用这些 Key；
- 不要把带 Key 的脚本随便发给别人；
- 如果换账号，就在 `start_realtime_human.sh` 里换成新的 Key。



##### 1.8 是否复制历史上传的头像和声音

历史上传文件在下面两个目录：

```bash
resource/avatar/flashhead/uploads
resource/voice/cosyvoice/uploads
```

如果希望新实例和旧实例完全一样，可以一起复制。

如果希望新实例是干净的，可以不复制这两个目录，或者复制后清空里面的文件。

##### 1.9 最简单检查清单

启动新实例前，按这个顺序检查：

```text
1. 新公网地址已经写进 start_realtime_human.sh
2. start_realtime_human.sh 里的 PORT 是 6006，或者是你自己指定的新端口
3. YAML 里的 service.port 和 PORT 一样
4. stop_realtime_human.sh 里的 PORT 和 CONFIG_FILE 也一致
5. AutoDL 或 nginx 已经把公网入口转到这个端口
6. TURN 配置决定好是继续用旧的，还是换新的
7. API Key 确认能用
8. 历史头像和音频文件确认要不要带过去
```

### 二、修改代码后如何 Git

本项目采用多人协作开发方式，主分支为 `clean-main`，每个开发者需要在自己的开发分支上进行修改，测试通过后再合并回主分支。

#### 基本信息

- **主分支**：`clean-main`
- **远程仓库**：`origin`
- **当前开发分支**：`XXX`

> 注意：`XXX` 不是固定分支名，而是“当前开发者自己的开发分支”的占位符。
> 每个人实际使用时，都要把 `XXX` 替换成自己的真实开发分支名。

例如：

- 如果你的开发分支是 `lgf-realtime`，那么所有 `XXX` 都要替换成 `lgf-realtime`；
- 如果你的开发分支是 `gry`，那么所有 `XXX` 都要替换成 `gry`。

---



#### 1. 从远程仓库获取最新信息

```bash
git fetch origin
```

说明：

`git fetch` 只会更新远程分支引用，例如 `origin/clean-main`，不会直接修改你当前本地分支的代码。

---



#### 2. 切换到本地主分支 `clean-main`

```bash
git checkout clean-main
```

---



#### 3. 同步远程主分支到本地

```bash
git pull --ff-only origin clean-main
```

说明：

`--ff-only` 表示只允许“快进式合并”，不会自动产生新的 merge commit。

如果这里报错，说明本地 `clean-main` 和远程 `clean-main` 出现了分叉，需要先人工确认，不要盲目强推。

---



#### 4. 切换回自己的开发分支

```bash
git checkout XXX
```

使用时需要把 `XXX` 替换成自己的真实开发分支名。

例如：

```bash
git checkout lgf-realtime
```

---



#### 5. 将最新主分支合并到自己的开发分支

```bash
git merge clean-main
```

这样做的目的是：

- 让自己的分支包含其他人已经合入主分支的最新代码；
- 提前发现并解决冲突；
- 保证后续合并回主分支时风险更小。

---



#### 6. 如果出现代码冲突

先查看冲突文件：

```bash
git status
```

手动解决冲突后，执行：

```bash
git add .
git commit
```

如果没有冲突，这一步可以跳过。

---



#### 7. 在自己的开发分支上测试

在合并回主分支之前，需要确认自己的分支可以正常运行。

例如：

```bash
python app.py
```

或者：

```bash
bash run.sh
```

或者：

```bash
npm run dev
```

只有测试成功后，才可以继续合并回主分支。

---



#### 8. 测试成功后，切换回主分支

```bash
git checkout clean-main
```

---



#### 9. 将自己的开发分支合并回主分支

```bash
git merge --ff-only XXX
```

说明：

`--ff-only` 表示只允许快进合并。

如果 `XXX` 已经包含了 `clean-main` 的最新提交，那么 `clean-main` 可以直接移动到 `XXX` 的最新提交位置。

这种方式可以保持 Git 历史比较干净。

使用时一定要把 `XXX` 替换成自己的真实开发分支名。

例如：

```bash
git merge --ff-only lgf-realtime
```

---



#### 10. 推送主分支到远程仓库

```bash
git push origin clean-main:clean-main
```

说明：

这一步会把本地 `clean-main` 分支推送到远程仓库的 `clean-main` 分支。

---



#### 11. 如果合并回主分支时报错

如果执行下面命令时报错：

```bash
git merge --ff-only XXX
```

通常说明 `clean-main` 无法直接快进到 `XXX`，可能是因为：

- `clean-main` 又有了新的提交；
- 两个分支历史发生了分叉；
- 你的开发分支没有基于最新的 `clean-main`。

此时不要盲目强推，建议重新执行：

```bash
git fetch origin

git checkout clean-main
git pull --ff-only origin clean-main

git checkout XXX
git merge clean-main
```

解决冲突并测试成功后，再重新合并回 `clean-main`。

---



#### 完整命令汇总

下面是完整流程。使用时需要把 `XXX` 替换成自己的真实开发分支名。

```bash
# 1. 获取远程最新信息
git fetch origin

# 2. 切换到本地主分支
git checkout clean-main

# 3. 同步远程主分支
git pull --ff-only origin clean-main

# 4. 切换回自己的开发分支
git checkout XXX

# 5. 将最新主分支合并到自己的开发分支
git merge clean-main

# 6. 如果有冲突，解决冲突后提交
# git status
# git add .
# git commit

# 7. 在自己的开发分支上测试
# python app.py
# bash run.sh
# npm run dev

# 8. 测试成功后，切换回主分支
git checkout clean-main

# 9. 将自己的开发分支快进合并回主分支
git merge --ff-only XXX

# 10. 推送到远程 clean-main 分支
git push origin clean-main:clean-main
```

---



#### 注意事项

1. 不要直接在 `clean-main` 上开发。
2. 每个人都应该使用自己的开发分支。
3. 合并回主分支前必须先测试。
4. 不要随意使用 `git push --force`。
5. `XXX` 必须替换成自己的真实开发分支名。
